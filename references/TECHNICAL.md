# Tasker 案件搜尋 - 技術細節

## 設計決策

### 為什麼選擇頁面搜尋框自動化？
1. **穩定性高**: 模擬用戶真實行為，即使網站更新也相對穩定
2. **可重用性**: 與現有登入邏輯一致，可重用 DynamicSession
3. **不依賴 URL**: 不依賴特定的 URL 結構，減少維護成本

### 為什麼關鍵字預設為空？
1. **靈活性**: 允許用戶獲取所有最新案件，不限制於特定關鍵字
2. **使用便利**: 常見的使用場景是查看最新案件，不一定需要篩選
3. **向下兼容**: 保持與現有功能的兼容性

### 為什麼輸出到終端機？
1. **Agent 使用**: 主要為了讓 AI Agent 使用，輸出到終端機更靈活
2. **管道支援**: 可以與其他工具結合使用（例如：grep、less）
3. **簡化**: 不需要處理文件寫入權限和路徑問題

## 實現細節

### 搜尋流程
1. **登入**: 使用記住我功能，避免重複登入
2. **載入頁面**: 載入案件頁面
3. **搜尋**: 如果有關鍵字，填寫搜尋框並提交
4. **提取連結**: 從頁面提取所有案件連結
5. **訪問詳情**: 逐一訪問案件詳細頁面
6. **提取資料**: 使用 CSS 選擇器提取案件資訊
7. **排序**: 按更新時間排序
8. **選擇**: 選擇前 N 筆案件
9. **輸出**: 格式化為 Markdown 並輸出

### 時間解析
目前時間解析是簡化的實現，實際使用時需要根據網站的時間格式調整。

常見時間格式：
- `2026-05-04 14:30`
- `2026/05/04`
- `2小時前`
- `今天 14:30`

### 錯誤處理策略
- **網路錯誤**: 記錄錯誤，終止程式
- **頁面載入失敗**: 記錄警告，跳過該頁面
- **提取失敗**: 記錄警告，使用預設值
- **排序失敗**: 記錄警告，將該案件排在最後

## 效能優化

### 現狀
- 串行訪問每個案件頁面
- 每個頁面載入約 5-10 秒
- 5 筆案件約 30-60 秒

### 優化方向
1. **並行訪問**: 使用多線程或異步 I/O
2. **快取**: 對已訪問的頁面進行快取
3. **增量更新**: 只提取新增的頁面

## 測試策略

### 單元測試
測試各個獨立的功能：
- 關鍵字合併邏輯
- 時間解析功能
- CSS 選擇器提取
- Markdown 格式化

### 整合測試
測試完整的搜尋流程：
- 端到端搜尋
- 不同的關鍵字組合
- 不同的 top 數值

### 邊界測試
測試邊界情況：
- 空關鍵字
- 沒有搜尋結果
- top=0 和 top=100
- 網路失敗

## 安全考慮

### 敏感資訊
- 登入憑證存儲在 `.env` 文件
- `.env` 文件已加入 `.gitignore`
- 持久化數據存儲在 `/tmp` 目錄

### 權限
- 確保腳本不會訪問未授權的資源
- 遵守網站的使用條款
- 適度的請求頻率
